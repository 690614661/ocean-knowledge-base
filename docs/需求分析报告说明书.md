## 1. 文档概述

### 1.1 编写目的

本文档旨在系统性地梳理海洋生物知识库（Ocean Knowledge Base）的可扩展方向，对功能需求进行分级、分类和详细描述，为后续的迭代开发提供决策依据和设计参考。

### 1.2 适用范围

本文档面向项目决策者、开发团队及产品设计人员，涵盖后端（Spring Boot）、前端（Vue.js）及基础设施层面的功能扩展需求。

### 1.3 术语与缩写

| 术语 | 说明 |
|------|------|
| ES | ElasticSearch，全文检索引擎 |
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| OSS | Object Storage Service，对象存储服务（本项目采用七牛云） |
| SSE | Server-Sent Events，服务端推送事件 |
| WebSocket | 全双工通信协议 |
| CRUD | 增删改查操作 |

## 4. 第一梯队功能详细需求

### F-01 搜索接入ES（笔记全文搜索）

#### 现状

- `NoteService.publicList()`：笔记搜索使用 `LIKE %keyword%` 模糊匹配标题，不支持全文检索
- `SearchController`：文档搜索通过 RestTemplate 直连ES，逻辑写在Controller中

#### 目标

将笔记搜索改造为基于 ElasticSearch 的全文检索，支持标题+内容匹配和高亮显示。

#### 功能性需求

| 需求ID | 描述 | 优先级 |
|--------|------|--------|
| F01-01 | 笔记保存/编辑时同步到ES `note_index` 索引 | P0 |
| F01-02 | 笔记删除时同步删除ES索引 | P0 |
| F01-03 | 支持按 `title` + `content` 多字段匹配搜索 | P0 |
| F01-04 | 搜索结果高亮匹配词（标题和内容） | P0 |
| F01-05 | 搜索范围限定为公开笔记（`is_public=1`） | P0 |
| F01-06 | 分页参数（page/size）与文档搜索保持一致 | P1 |
| F01-07 | 搜索词联想/自动补全（建议输入3个字符后触发） | P2 |

#### 非功能需求

| 需求ID | 描述 | 指标 |
|--------|------|------|
| F01-N01 | ES写入异步化，不影响笔记保存响应时间 | < 200ms |
| F01-N02 | ES搜索响应时间 | < 500ms |
| F01-N03 | ES索引最终一致性（允许秒级延迟） | ≤ 3秒 |
| F01-N04 | ES不可用时降级回数据库LIKE搜索 | 静默降级 |

#### ES索引映射设计

```json
{
  "note_index": {
    "settings": {
      "analysis": {
        "analyzer": {
          "ik_smart_analyzer": { "type": "custom", "tokenizer": "ik_smart" }
        }
      }
    },
    "mappings": {
      "properties": {
        "noteId":     { "type": "long" },
        "userId":     { "type": "long" },
        "title":      { "type": "text", "analyzer": "ik_max_word" },
        "content":    { "type": "text", "analyzer": "ik_max_word" },
        "isPublic":   { "type": "byte" },
        "viewCount":  { "type": "integer" },
        "voteCount":  { "type": "integer" },
        "createTime": { "type": "date", "format": "yyyy-MM-dd HH:mm:ss" },
        "updateTime": { "type": "date", "format": "yyyy-MM-dd HH:mm:ss" }
      }
    }
  }
}
```

#### 受影响文件清单

| 文件 | 改动类型 | 说明 |
|------|----------|------|
| `NoteService.java` | 修改 | 新增 `syncNoteToEs()`, `deleteNoteFromEs()` 方法 |
| `NoteController.java` | 修改 | 搜索接口改为调用ES（或新增统一搜索端点） |
| `SearchService.java` | **新增** | 统一搜索服务（文档+笔记联合搜索） |
| `SearchController.java` | 重构 | 剥离逻辑到 SearchService |
| `ElasticsearchInitConfig.java` | 修改 | 增加 `note_index` 初始化 |

#### 验收标准

1. [ ] 新建公开笔记后，ES中可检索到（≤3秒延迟）
2. [ ] 搜索关键词匹配笔记标题或内容，结果正确
3. [ ] 搜索结果高亮显示匹配词
4. [ ] 私有笔记不出现在搜索结果中
5. [ ] 删除笔记后ES索引同步删除
6. [ ] ES不可用时，搜索功能降级可用

### F-02 抽取 SearchService

#### 现状

`SearchController.java` 中直接构建ES查询JSON、调用RestTemplate、解析ES响应，全部逻辑耦合在Controller中，约120行代码。

#### 目标

抽取独立的 `SearchService` 层，将ES操作封装到Service中，Controller只做参数校验和响应封装。

#### 设计

```
Controller层
  SearchController.search(params)
        │
        ▼
Service层
  SearchService.search(keyword, page, size, filters)
        │
        ├── buildQueryJson()          // 构建ES查询
        ├── executeSearch()           // 调用ES HTTP API
        ├── parseResponse()           // 解析ES响应
        └── buildHighlightResult()    // 处理高亮
        │
        ▼
Mapper层  (无，直接通过RestTemplate调用ES)
```

#### 接口定义

```java
public interface SearchService {

    /**
     * 全文检索
     * @param keyword 搜索关键词
     * @param page 页码（从1开始）
     * @param size 每页条数
     * @param filters 过滤条件（分类、电子书ID、日期范围等）
     * @return 搜索结果
     */
    SearchResult search(String keyword, int page, int size, SearchFilter filters);

    /**
     * 同步文档到ES索引
     */
    void syncDocIndex(Long docId, String name, String content, Long ebookId);

    /**
     * 同步笔记到ES索引
     */
    void syncNoteIndex(Long noteId, String title, String content, Long userId, boolean isPublic);

    /**
     * 从ES删除索引
     */
    void deleteIndex(String indexName, Long id);
}
```

#### 验收标准

1. [ ] 搜索功能功能无变化，返回数据结构兼容旧版前端
2. [ ] 可以独立为 SearchService 编写单元测试
3. [ ] ES同步/删除逻辑复用统一方法

### F-06 RAG 检索增强生成 ⭐🔥

#### 现状

AI 对话的 system prompt 明确指出："Your knowledge comes from training data, not from knowledge base document retrieval."（你的知识来自训练数据，不基于知识库内文档检索。）

#### 目标

引入 RAG 架构，让 AI 在回答前先检索知识库中的相关文档/笔记，将检索结果作为上下文注入 prompt，实现基于知识库内容的精准回答。

#### 架构设计

```
用户提问
   │
   ▼
┌─────────────────────┐
│  1. 检索知识库       │  ← 调用 SearchService 搜索文档+笔记
│  (ES向量/全文检索)    │
└─────────┬───────────┘
          │ 返回 Top-K 相关片段
          ▼
┌─────────────────────┐
│  2. 构建增强 Prompt  │  ← system prompt + 检索到的文档片段 + 对话历史
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  3. 调用 AI 生成回答  │  ← 大模型基于上下文回答问题
│  (DeepSeek/百炼)     │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  4. 返回回答+引用来源 │  ← 显示参考了哪些文档
└─────────────────────┘
```

#### Prompt 模板设计

```
你是"海洋知识库"的 AI 助手，专注于海洋生物领域。
请基于以下知识库内容回答用户问题。
如果你在知识库中找到了相关信息，请优先采用并标注引用来源。
如果知识库中没有足够信息，可以结合你的知识回答，但要明确说明"知识库中未找到相关信息"。

【相关文档】
1. 《海洋鱼类大全》> 鲨鱼分类
   鲨鱼属于软骨鱼纲，分为...（文档ID: 12345）
   
2. 《海洋生态》> 食物链
   海洋食物链的顶端是...（文档ID: 67890）

【用户问题】
鲨鱼和鲸鱼有什么区别？
```

#### 功能需求

| 需求ID | 描述 | 优先级 |
|--------|------|------|
| F06-01 | AI对话时自动检索Top-5相关文档片段作为上下文 | P0 |
| F06-02 | 回答底部展示引用来源（文档名称 + 链接） | P0 |
| F06-03 | 知识库无相关内容时明确提示用户 | P1 |

public interface FileStorageStrategy {
    
    /** 上传文件 */
    String upload(byte[] data, String fileName);
    
    /** 删除文件 */
    void delete(String fileName);
    
    /** 获取完整可访问 URL */
    String getUrl(String fileName);
}

**七牛云实现**（生产环境使用）：

```java
/**
 * 七牛云对象存储实现
 * 
 * 使用步骤：
 * 1. 七牛云控制台创建 Bucket（存储空间）
 * 2. 获取 Access Key / Secret Key（个人中心 → 密钥管理）
 * 3. 绑定 CDN 加速域名（可选，推荐）
 * 4. 配置 application.yml 中 ocean.file.qiniu.*
 */
@Component
@ConditionalOnProperty(value = "ocean.file.storage-type", havingValue = "qiniu")
public class QiniuFileStorage implements FileStorageStrategy {

    @Value("${ocean.file.qiniu.access-key}")
    private String accessKey;

    @Value("${ocean.file.qiniu.secret-key}")
    private String secretKey;

    @Value("${ocean.file.qiniu.bucket}")
    private String bucket;

    @Value("${ocean.file.qiniu.domain}")
    private String domain;

    private UploadManager uploadManager;
    private Auth auth;
    private BucketManager bucketManager;

    @PostConstruct
    public void init() {
        Configuration cfg = new Configuration(Region.autoRegion());
        uploadManager = new UploadManager(cfg);
        auth = Auth.create(accessKey, secretKey);
        bucketManager = new BucketManager(auth, cfg);
    }

    /**
     * 上传文件到七牛云
     * @param data     文件字节数组
     * @param fileName 文件名（含目录，如 cover/ebook_xxx.jpg）
     * @return 完整可访问URL
     */
    @Override
    public String upload(byte[] data, String fileName) {
        String upToken = auth.uploadToken(bucket);
        try {
            Response response = uploadManager.put(data, fileName, upToken);
            // 解析上传结果
            DefaultPutRet putRet = new Gson().fromJson(
                response.bodyString(), DefaultPutRet.class);
            return domain + "/" + putRet.key;
        } catch (QiniuException ex) {
            throw new BusinessException("文件上传到七牛云失败: " + ex.getMessage());
        }
    }

    /**
     * 从七牛云删除文件
     */
    @Override
    public void delete(String fileName) {
        try {
            bucketManager.delete(bucket, fileName);
        } catch (QiniuException ex) {
            // 文件不存在时忽略
            if (ex.code() != 612) {
                throw new BusinessException("删除文件失败: " + ex.getMessage());
            }
        }
    }

    @Override
    public String getUrl(String fileName) {
        return domain + "/" + fileName;
    }
}
```

引入 AI 多模态能力，支持上传海洋生物图片进行识别/分析，以及根据文本描述生成海洋生物相关图片。使知识库具备"看图识物"和"文生图"的能力。

#### 架构设计

```
┌─────────────────────────────────────────────┐
│              两种多模态场景                    │
└─────────────────────┬───────────────────────┘
                      │
        ┌─────────────┴─────────────┐
        ▼                           ▼
┌─────────────────┐     ┌─────────────────────┐
│  场景A：图像识别  │     │  场景B：图像生成      │
│  (图片→文本)      │     │  (文本→图片)         │
├─────────────────┤     ├─────────────────────┤
│ 用户上传图片      │     │ 用户输入描述文本      │
│    ↓             │     │    ↓                │
│ 调用视觉API      │     │ 调用文生图API        │
│  (识别海洋生物)    │     │  (生成海洋生物图片)   │
│    ↓             │     │    ↓                │
│ 返回识别结果      │     │ 返回图片URL          │
│  + 关联知识库文档  │     │  + 可收藏/下载       │
└─────────────────┘     └─────────────────────┘
```

#### 功能需求

| 需求ID | 描述 | 优先级 |
|--------|------|--------|
| F16-01 | **海洋生物图片识别**：用户上传海洋生物图片，AI识别物种名称、分类、特征 | P1 |
| F16-02 | **识别结果 + 知识库关联**：识别后自动搜索知识库中相关文档，提供知识链接 | P1 |
| F16-03 | **图片识别对话**：在AI对话中上传图片，AI理解图片内容并回答相关问题 | P1 |
| F16-04 | **海洋生物图片生成**：输入文字描述，AI生成对应的海洋生物插画/示意图 | P2 |
| F16-05 | **图片生成记录**：生成的图片保存至用户个人画廊，可查看历史 | P2 |
| F16-06 | **批量识别**：一次上传多张图片，批量识别海洋生物 | P2 |

#### 多模态能力矩阵

| 外部API | 能力 | 适用场景 | 成本 |
|---------|------|----------|------|
| **阿里云百炼（通义千问VL）** | 图片理解、视觉问答 | 海洋生物图片识别 | 已集成百炼，扩展成本低 |
| **DeepSeek（未开源视觉版）** | 暂不支持视觉 | — | — |
| **OpenAI GPT-4o/GPT-4V** | 图片理解+识别 | 高精度识别 | 需额外接入，$0.01+/张 |
| **阿里云视觉智能平台** | 动物识别专用API | 物种精准鉴定 | 按调用量计费 |
| **百度文心一言ERNIE-Bot** | 图片理解 | 可作备选 | API独立接入 |
| **Stable Diffusion（本地/API）** | 文生图 | 海洋生物插画生成 | 本地部署or API |

#### 推荐技术选型

```
识别场景：
  主方案：阿里云百炼（通义千问VL）—— 与现有 AI 供应商一致，统一管理
  备选：OpenAI GPT-4o（如需更高识别精度）

生成场景：
  推荐：Stable Diffusion API / 阿里云百炼（通义万相）
  说明：文生图需求低频，按需调用，不需要本地部署
```

#### Prompt 模板设计（图片识别场景）

```
你是"海洋知识库"的 AI 助手，专注于海洋生物领域。
用户上传了一张海洋生物图片，请：
1. 识别图片中的海洋生物
2. 给出该生物的：
   - 中文名/学名
   - 分类（门/纲/目/科/属）
   - 主要特征
   - 分布区域
   - 生态习性
3. 如果无法确定物种，给出可能的分类范围
4. 始终保持专业、准确
```

#### 数据库设计（图片生成记录）

```sql
CREATE TABLE IF NOT EXISTS `ai_image_record` (
    `id` BIGINT NOT NULL COMMENT '记录ID（雪花算法）',
    `user_id` BIGINT NOT NULL COMMENT '用户ID',
    `type` VARCHAR(20) NOT NULL COMMENT '类型：recognition(识别) / generation(生成)',
    `original_image_url` VARCHAR(500) DEFAULT NULL COMMENT '原图URL（识别场景）',
    `result_image_url` VARCHAR(500) DEFAULT NULL COMMENT '结果图URL（生成场景）',
    `prompt_text` TEXT DEFAULT NULL COMMENT '用户输入的文本提示词',
    `result_text` TEXT DEFAULT NULL COMMENT '识别结果文本',
    `species` VARCHAR(100) DEFAULT NULL COMMENT '识别的物种名称',
    `provider` VARCHAR(20) NOT NULL COMMENT 'AI供应商',
    `model` VARCHAR(50) NOT NULL COMMENT '模型名称',
    `cost_yuan` DECIMAL(10,6) NOT NULL COMMENT '费用（元）',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    PRIMARY KEY (`id`),
    KEY `idx_user_id` (`user_id`),
    KEY `idx_type` (`type`),
    KEY `idx_species` (`species`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='AI图像记录表';
```

#### 注意事项 / 约束

1. **成本控制**：视觉API调用成本是文本的 5-20 倍，需限制单用户日调用次数
2. **内容安全**：上传图片需做内容审核（黄暴恐过滤），可使用阿里云内容安全服务
3. **合规性**：生成的图片需加平台水印，避免滥用
4. **存储**：上传的图片和生成的图片通过 F-11 的 FileStorageStrategy 统一存入七牛云（或本地），目录统一为 `ai/recognize/` 和 `ai/generate/`
5. **识别精度**：AI识别不能保证100%准确，结果页需标注"AI识别结果，仅供参考"

#### 验收标准

1. [ ] 上传一张海龟图片，AI能正确识别出物种名称及分类

3. [ ] 在AI对话中上传图片，AI能基于图片内容回答问题
4. [ ] 输入"一只发光的深海鮟鱇鱼"，生成对应的图片
5. [ ] 用户画廊页面展示历史识别和生成记录
6. [ ] 超限制调用时提示"今日调用次数已达上限"
7. [ ] 上传违规图片时被拦截并提示

## 6. 技术债务修复建议

### T-01 快照频率调优

```yaml
# 当前：每分钟执行
# 建议：每5分钟执行，减少数据库压力
ocean:
  snapshot:
    cron: "0 */5 * * * ?"  # 每5分钟
```

### T-02 Fastjson → Jackson 迁移

Fastjson 1.2.83 虽然已修复已知漏洞，但社区活跃度下降，项目同时在使用 Jackson 和 Fastjson，建议统一为 Jackson。

**迁移计划：**
1. 搜索所有 `import com.alibaba.fastjson` 引用
2. 逐个替换为 Jackson 的 `ObjectMapper`
3. 移除 `pom.xml` 中的 fastjson 依赖

**当前引用位置：**
| 文件 | 行数 | 用途 |
|------|------|------|
| `SearchController.java` | 68 | `JSON.parseObject()` |
| `AiService.java` | 多行 | `JSON.parseObject()`, `JSONObject` |
| 其他 | 待排查 | 待排查 |

### T-03 缓存抽象化（建议后续迭代做）

**目标：** 将散落在各Service的 Redis 操作统一为 Spring Cache 注解。

**涉及缓存点：**
| Service | 缓存Key | TTL |
|---------|---------|-----|
| `EbookService` | `ebook:list:page:1` | 1小时 |
| `DocService` | `doc:tree:{ebookId}` | 5分钟 |
| 点赞防重 | `doc_vote:{docId}:{ip}:{date}` | 24小时 |
| 点赞防重 | `note_vote:{noteId}:{ip}:{date}` | 24小时 |
| 阅读历史 | `history:user:{userId}` | 30天 |
| 限流 | `rate_limit:{ip}:{api}` | 1分钟/1天 |

### T-04 单元测试建设

**建议策略：**
1. 优先为核心 Service（`DocService`、`SearchService`、`AiService`）编写单元测试
2. 使用 Mockito 模拟 Mapper 和外部依赖
3. 使用 `@WebMvcTest` 测试 Controller 层
4. 测试覆盖率目标：核心模块 ≥ 60%
